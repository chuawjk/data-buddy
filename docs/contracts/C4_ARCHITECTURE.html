Level 3

Component diagrams

Each container decomposed into its internal components. Color encodes role: purple = frontend-facing, coral = coordination/entry, teal = supporting services, amber = core runtime, gray = infrastructure or external reference. Dashed borders indicate external containers shown for context.

Backend service

Three tiers: a frontend-facing layer (HTTP router + SSE proxy), a coordination layer (stage orchestrator flanked by state manager, prompt library, and watchdog), and an OpenCode interface layer. The stage orchestrator is the central coordinator — it calls the prompt library to build prompts, tells the state manager when to persist, registers the watchdog per turn, and hands the OpenCode client the prompt to fire.

HTTP routerFastAPI / Express REST endpoints Exposes all frontend-facing REST routes: GET /state, POST /turn, POST /section/:id/accept, POST /section/:id/drop, POST /plan/update, GET /export. Delegates business logic to the stage orchestrator.
SSE proxyAsync generator Event forwarding Holds the frontend SSE connection. Receives enriched events from the stage orchestrator (which filters and annotates raw OpenCode events) and streams them to the browser. Handles client reconnection without losing events mid-turn.
Stage orchestratorPython class Core coordination Central coordinator. Knows the current stage and what action is required next. On a user action: reads state via the state manager, selects the right prompt template from the prompt library, fires the turn via the OpenCode client, registers the watchdog, and forwards incoming SSE events to the SSE proxy. On backend-only operations: delegates directly to the state manager.
State managerFile I/O Persistence All reads and writes of state.json. Owns the schema: stage, aim, OpenCode session ID, plan array, section statuses, profile summary. Provides atomic write (write-to-temp, rename) to avoid partial reads during an active agent turn.
Prompt libraryTemplate strings Prompt templates One template per agent-driven operation: profile, plan, section build (N), section redirect, plan revision. Each template is a function that accepts context (aim, profile JSON, plan JSON, section metadata) and returns the parts and format payload for prompt_async.
WatchdogAsync timer Liveness monitoring Started at the beginning of each agent turn, reset on every SSE event (including heartbeats and reasoning deltas). Fires after 60s of silence: calls POST /session/:id/abort via the OpenCode client. If the session doesn't return to idle within 10s, creates a fresh OpenCode session and updates state.json.
OpenCode clientHTTP + SSE OpenCode interface Wraps all OpenCode HTTP calls. Holds the single persistent SSE subscription to GET /event (ADR-006). Filters events by current session ID and forwards them to the stage orchestrator. Reconnects automatically if the heartbeat stops. Exposes create_session(), prompt_async(), and abort().

React SPA

Intentionally lean per ADR-007. Two backend-facing components at the top (API client for REST, SSE handler for the live stream), three rendering components below. There is no client-side state store — stage views re-hydrate from GET /state on every page load, and the SSE handler drives live updates only during active agent turns.

HTTP APIopencode serve Official docs OpenAPI 3.1 server at localhost:4096. Key endpoints: POST /session, POST /session/:id/prompt_async (returns 204 immediately), GET /event (SSE stream), POST /session/:id/abort. Spec served at /doc.
Session managerSQLite / Drizzle DeepWiki / spike Persists sessions, messages, and parts to ~/.local/share/opencode/opencode.db (WAL mode). Sessions survive server restarts. History is re-loaded from the DB on each new turn — the caller does not need to replay it.
Event busInternal pub/sub Official docs / spike All internal components publish to this bus. The HTTP API subscribes and forwards over SSE as {"type": "...", "properties": {...}} events. Heartbeat fires every ~10s. Confirmed working on v1.15.10; known regression in v1.14.42–46 (issue #26697).
Agent loopInternal DeepWiki Runs the LLM turn: builds the context window (system prompt + history + tools), calls the model, iterates over tool calls until the model stops. Each tool call dispatches to the tool executor and emits events on the bus. May spawn sub-agent sessions via the Task tool.
Compaction managerInternal Official docs / issue #11314 Fires automatically when token usage approaches ~75% of the model's context window. Summarises old turns and prunes tool outputs. Emits a session.compacted event and a CompactionPart. Configurable via compaction.auto in opencode.json.
Tool executorInternal Spike / official docs Runs built-in tools: apply_patch (unified diff — confirmed as the file-write tool in the spike), bash, read. Each tool call emits message.part.updated with status: pending → running → completed. apply_patch results include metadata.files[] with per-file diffs and line counts.
Level 4

Code diagrams

Implementation detail for the three highest-leverage components: the state.json data contract, the stage orchestrator's state machine, and the OpenCode client's SSE connection lifecycle. These are the pieces a Python developer needs to see before writing the first line of code.

state.json schema

The single source of truth for the entire brief. Written atomically by the state manager (write to state.tmp.json, rename to state.json) to prevent partial reads during active agent turns. The SPA reads it wholesale via GET /state on every page load.

Stage orchestrator — state machine

The orchestrator is a state machine at heart. Five stable states: idle, profiling, planning, building, done. The redirecting sub-state sits inside building and doesn't change the section pointer. All three agent-driven states share a single failed error state — retry re-fires the same prompt for whichever state failed.

OpenCode client — SSE connection lifecycle

The concrete implementation of ADR-006. One GET /event connection opened at startup and held for the lifetime of the process. Every prompt_async call fires and returns immediately (204); the already-open stream delivers all subsequent events. Session ID filtering is the single line of logic that makes the persistent subscription safe across multiple turns.

Architecture documentation complete through L4. All decisions recorded in the ADR. Ready to begin implementation.
  • Start with the OpenCode client and a working state.json read/write cycle
  • Then the stage orchestrator state machine — the profiling stage first as it's the simplest end-to-end path
  • Frontend last — it has the least uncertainty once the backend API surface is stable
