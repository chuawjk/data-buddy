# Plan: N2-S11 — Drop section

**Date:** 2026-06-03
**Story:** N2-S11 · Drop section (S)

## What this does

Implements `POST /section/:id/drop` — marks a proposed section as dropped in
`state.json` and returns 204. No OpenCode call is made. Dropped sections are
already excluded from export by `GET /export` (which filters for `status=="accepted"`).

## Files to create/modify

- `backend/router.py` — replace the stub `post_section_drop` handler with real logic
- `backend/tests/unit/app/test_section_drop.py` — new test file (TDD)

## Approach

Same pattern as N2-S10 (accept section):
1. Read `state_manager.get_state()["plan"]` to find the section by `section_id`.
2. If not found → 400 `section_not_found` error envelope.
3. If found but status is not `"proposed"` → 400 `section_not_proposed`.
4. Set `status = "dropped"` on the matching section.
5. `state_manager.update(plan=updated_plan)`.
6. Return 204.

Note: the API contract says "removes the section from the plan array", but the
story says "dropped sections excluded from export (already handled by GET /export)".
The export filters by status=="accepted", so setting status="dropped" is the correct
approach — it preserves the audit trail while excluding from export.
