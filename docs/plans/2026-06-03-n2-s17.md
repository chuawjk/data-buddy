## What this story does

Adds an Export button to the UI that calls `GET /api/export` and triggers a `.md` file download.
The button is disabled when no sections have been accepted; enabled as soon as one is.

## Design decision: where does the export control live?

**Decision: a standalone `ExportButton` component rendered in the App.tsx header/toolbar — not
inside `BuildView`.**

Rationale:

1. The mockup shows the export control in the top bar (`↓ Export` chip on the same row as the
   brand and aim, visible during the building/done stages — see Stage 4 and 4b mockups).
2. The export control is a cross-cutting action that sits above the stage view, not within it.
   Embedding it in `BuildView` would make it invisible if the component is ever split or if the
   done view diverges from the building view in Night 3.
3. `App.tsx` already tracks `stage` and `loading` state. Adding `acceptedSectionCount` to
   `AppState` is a small, natural extension; no new global state store is needed.
4. `ExportButton` needs only two props: `disabled: boolean` and nothing else (it fetches
   internally on click) — making it easy to render anywhere without coupling.

The button renders in the existing `<header>` bar in `App.tsx`, right-aligned. It is only
shown when `stage === "building" || stage === "done"`.

## Component structure

```
ExportButton
  props:
    disabled: boolean       — true when acceptedSectionCount === 0
  local state:
    isExporting: boolean    — true while GET /export is in flight
    error: string | null    — last error message (auto-cleared after 4s)
  behaviour:
    onClick → api.getExport() → create Blob → anchor.click() download → clear loading
    error path → set error message → 4s timeout → clear error
```

`App.tsx` additions to `AppState`:
- `plan: Section[]` — hydrated from `GET /state`; updated on `plan.ready` SSE event
- `acceptedSectionCount` derived inline: `state.plan.filter(s => s.status === "accepted").length`

---

## State management

**What stays local to `ExportButton`:**
- `isExporting` — spinner/disabled state while fetch is in flight
- `error` — inline error text, auto-dismissed

**What comes from App.tsx via props:**
- `disabled` — computed from `plan` array in App state; ExportButton does not know about sections

**What drives updates in App:**
- On `plan.ready` SSE event: update `state.plan` from `event.sections`
- On `stage.changed` SSE event: already handled; `plan` persists across stage changes
- On page load: `GET /state` already called; extend the parse to capture `plan`

The App.tsx SSE handler for `plan.ready` is new: currently App only handles `stage.changed`
and `profile.ready`. N2-S15 (PlanView) will also need plan state, but that is its own story;
here we only add what ExportButton needs.

---

## SSE events this component reacts to

ExportButton itself listens to no SSE events. The `disabled` prop is derived from `App.tsx`
state, which is updated when `plan.ready` fires.

The App.tsx SSE handler additions needed for N2-S17:
- `plan.ready` → update `state.plan` from `event.sections`
- (Stage.changed to "done" already handled; no plan-specific action needed there)

## Vitest test plan (`ExportButton.test.tsx`)

**Happy path:**
- Given `disabled={false}`, when rendered, then `export-btn` is present and not disabled.
- Given the button is clicked, when `api.getExport()` resolves with markdown text, then a
  download is triggered (mock `URL.createObjectURL` and `anchor.click`).

**Error paths:**
- Given the button is clicked, when `api.getExport()` rejects, then `export-error` appears with
  an error message.
- Given an error is shown, after 4 seconds, then `export-error` is removed.

**Edge cases / null inputs:**
- Given `disabled={true}`, when rendered, then `export-btn` has the `disabled` attribute.
- Given `disabled={true}`, when clicked, then `api.getExport()` is NOT called.
- Given a very long export markdown string, when download is triggered, then the Blob is
  created successfully (no truncation).

**App.tsx integration:**
- Given `GET /state` returns `plan: []`, when rendered, then `export-btn` is disabled (0 accepted).
- Given `GET /state` returns a plan with one `accepted` section, when rendered at `building`
  stage, then `export-btn` is enabled.
- Given a `plan.ready` SSE event arrives with sections containing an accepted section, when
  processed, then the button becomes enabled.

## Risks

1. **Download mechanism in test environment.** `URL.createObjectURL` and `document.createElement("a")` + `.click()` are browser-only. Vitest (jsdom) requires mocking these. The test plan uses `vi.stubGlobal("URL", ...)` or a similar pattern.
2. **Race condition on export error dismissal.** The 4-second auto-dismiss uses `setTimeout`. Tests must use `vi.useFakeTimers()` to assert the state change reliably.
3. **App.tsx plan state coupling.** N2-S15 (PlanView) will also need to read and update `plan` in App state. There is a risk of merge conflicts if N2-S15 lands first and adds `plan` differently. Mitigation: this plan proposes adding `plan: Section[]` to `AppState` — N2-S15 should use the same field. If N2-S15 merges first, N2-S17 will rebase and remove the duplicate addition.
4. **plan.ready SSE handler in App.tsx.** Currently the App.tsx SSE handler only reacts to `stage.changed` and `profile.ready`. Adding `plan.ready` here is minimal but could conflict with N2-S15 which will do the same. Resolution: whoever merges second wins; the two handlers are additive.
