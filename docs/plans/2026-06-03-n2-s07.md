# Plan: N2-S07 — Section build events

**Date:** 2026-06-03
**Branch:** feat/n2-s07-section-build-events
**Story:** N2-S07 — Section build events

## What

Translate the section turn's activity into domain events. Specifically:

1. **Activity events** (`tool.*`, `message.part`, `file.ready`) during a section turn already
   flow through `_normalise_and_publish` in `opencode_client.py`. These constitute the
   `section.building` activity stream. No new dispatch needed for activity events.

2. **`section.building` domain event** (SSE_CONTRACT.md §2.1) is emitted by the orchestrator
   immediately after dispatching the section build prompt via `start_build_section`.

3. **`session.idle` dispatch for building stage** — extend `start_bus_listener` to call
   `_handle_section_idle` when stage=building. Parallel to `_handle_profile_idle` /
   `_handle_plan_idle`.

4. **`section.proposed`** — emitted when all three triplet files (`.py`, `.png`, `.md`) are
   present at session.idle during building stage.

5. **`section.failed`** — emitted when files are missing at session.idle. (N2-S08 refines
   failure detection further.)

## Architecture boundary

The task says "primarily opencode_client.py" for idle dispatch. However the hard boundary:
- client never imports orchestrator
- client never accesses the event bus directly
- idle dispatch (reading state, publishing domain events) belongs in orchestrator

`_handle_profile_idle` and `_handle_plan_idle` both live in `orchestrator.py` — `_handle_section_idle` follows the same pattern. The client publishes `session.idle` to the bus; the orchestrator's bus listener picks it up.
