## Context

Post-Night-2, the orchestrator already handles auto-sequencing through queued sections (the
`_start_next_queued_section` helper introduced in post-N2 fixes). What is missing:

1. **N3-S01 — `done` state:** When the last section is accepted and there are no more `queued`
   sections to start, the orchestrator must transition to `stage=done` and emit
   `stage.changed(done)`. Currently `_start_next_queued_section` silently returns with
   `"all sections complete"` but never fires the `done` transition.

2. **N3-S01 — dropped-section skip:** The sequencing already skips dropped sections (only advances
   on `queued`) but the `done` detection needs to account for dropped sections — all non-dropped
   sections done means brief is complete.

3. **N3-S01 — rapid-accept re-entrancy guard:** If two `POST /section/:id/accept` calls land
   within the same event loop turn, there must be no double-trigger of the same next section.

4. **N3-S04 — watchdog per-turn reset:** For multi-section runs the watchdog must be fully
   cancelled after each section completes and re-armed with the correct timeout for the next.
   The current `cancel()` call in `_handle_section_idle` is correct but needs verification that
   a watchdog-triggered session swap persists the new session ID correctly in state so the next
   section turn targets the fresh session.

5. **N3-S04 — session-ID swap on stall:** When the watchdog fires mid-section-build, it replaces
   `opencode_session_id` in `state.json`. The next `start_build_section` call must pick up this
   new ID from state (it already reads from `state.get("opencode_session_id")` at call time, which
   is correct). We need to confirm no corruption of the plan or the section sequence.

## Implementation approach

### N3-S01 — `_start_next_queued_section` changes

The method currently returns early with a log when no queued section exists. The change:

1. After the `next_section is None` check, inspect the full plan:
   - Count sections that are neither `dropped` nor `done` (i.e., not `queued`, `building`). If
     every non-dropped section is `accepted`, then all sections are done.
   - If so: call `self._state_manager.update(stage="done")` and `await self._bus.publish("stage.changed", {"stage": "done", ...})`.
   - Emit with a timestamp (`ts = int(time.time() * 1000)`).

2. Dropped sections: The loop only starts `queued` sections. Dropped sections sit at `dropped`
   status and are never advanced to `queued`, so they are already skipped by the `next(...status == "queued")` scan. The `done` detection filters them out: we reach done when no `queued` sections remain AND all non-dropped sections are in a terminal state (`accepted` or `failed`).

   Concrete done condition: `all(s["status"] in {"accepted", "dropped", "failed"} for s in plan)`.

3. Re-entrancy guard: `_start_next_queued_section` reads fresh state on every call, so it is
   naturally idempotent as long as concurrent `accept_section` calls don't interleave between the
   `get_state()` read and the `start_build_section` dispatch. The guard: add a module-level (or
   instance-level) `asyncio.Lock` on `_start_next_queued_section` so concurrent calls are
   serialised. After taking the lock, re-read state, then proceed. This prevents double-trigger.

   The lock is an instance attribute `_section_loop_lock: asyncio.Lock` initialised in `__init__`.

4. Calling `accept_section` on an unknown ID already returns an error envelope from the router
   (N2-S10 existing code). The section loop itself is only triggered from `_handle_section_idle`
   after `session.idle`, which is not concurrent with itself.

### N3-S04 — watchdog per-turn timeout preservation

Currently `heartbeat()` calls `start_turn()` with no timeout, which defaults to `WATCHDOG_TIMEOUT`
(60s). For section builds the initial `start_turn(timeout=180)` is correct, but each heartbeat
resets the timer back to 60s. The fix:

- Track the currently active timeout in `self._current_timeout: int` on the `Watchdog` instance.
- `start_turn(timeout=...)` stores the value before creating the task.
- `heartbeat()` calls `start_turn(timeout=self._current_timeout)` to preserve the original timeout.
- `cancel()` clears `_current_timeout` back to `WATCHDOG_TIMEOUT` so the next `start_turn()` call
  without an explicit timeout uses the production default.

This is a small but correctness-critical change: without it a 180s section build watchdog would be
reset to 60s on the first heartbeat.

### N3-S04 — session swap persistence test

The existing `test_watchdog.py` already covers the happy path. The new tests verify:
- After `_handle_timeout`, `state_manager.update(opencode_session_id=new_id)` is called.
- On the subsequent `start_build_section`, the session_id is read fresh from state (already the
  case — `state.get("opencode_session_id")` is called inside `start_build_section`).
- `state.json` plan is NOT mutated by the watchdog (watchdog only touches `opencode_session_id`).

## Test plan (TDD)

### `test_section_loop_done.py` — N3-S01 criteria

1. **Happy path — last section accepted triggers `done`:** Plan with 2 sections both accepted.
   After `_start_next_queued_section` finds no queued sections and all are accepted, `stage.changed`
   with `stage=done` is emitted and `state.stage == "done"`.

2. **Dropped sections skipped — `done` when all non-dropped are accepted:** Plan with 3 sections:
   two accepted, one dropped. `_start_next_queued_section` emits `stage.changed(done)`.

3. **Still queued sections — no `done` yet:** Plan with one accepted, one queued. Method starts
   the next section (no `done` emission).

4. **Rapid-accept re-entrancy:** Two concurrent calls to `_start_next_queued_section` on a plan
   with one queued section. Only one `start_build_section` dispatch occurs.

5. **Null / edge — all sections dropped:** Plan with all sections dropped. `done` is emitted.

6. **Error path — `failed` sections count towards done:** Plan with one accepted, one failed,
   none queued. `done` is emitted.

### `test_watchdog_hardening.py` — N3-S04 criteria

1. **Per-turn timer reset — custom timeout preserved through heartbeat:** Start watchdog with
   `timeout=180`. Call `heartbeat()`. Verify the next watch task uses 180s, not 60s.

2. **Session swap persists to state, plan untouched:** After `_handle_timeout`, state update
   touches only `opencode_session_id`. Plan in state remains unchanged.

3. **Next section turn after session swap targets new session:** Simulate `_handle_timeout` then
   `start_build_section`. Client prompt is called with the new session ID.

4. **cancel() resets to default timeout:** After `cancel()`, a new `start_turn()` without an
   explicit timeout uses `WATCHDOG_TIMEOUT`.

## Risks / open questions

- **Concurrent `done` emission:** If `_handle_section_idle` fires while the orchestrator is
  already in `done` stage (e.g., a delayed `session.idle` from the last section), the
  `_start_next_queued_section` guard will re-check state and find no queued sections — but it
  should also guard against emitting `done` when `stage` is already `done`. I will add this guard.

- **`failed` sections and `done`:** Story says "dropped sections skipped". `failed` sections are
  not explicitly mentioned in the `done` detection. I will treat `failed` as terminal and include
  it in the done condition (alongside `accepted` and `dropped`), consistent with the UI design
  where a failed section with no user action still sits at `failed` and the loop should be able to
  terminate. If TL disagrees, this can be changed in a single-line fix.

- **Re-entrancy lock scope:** The lock is only needed for `_start_next_queued_section`. The lock
  will be acquired there and released before any awaits that could take long (we release before the
  `start_build_section` call to avoid holding the lock across the OpenCode round-trip).

- **No changes to `docs/contracts/`:** The `stage.changed` event with `stage=done` is already in
  `API_CONTRACT.html` as a valid payload. No contract changes needed.
