## What this story does

Builds out `PlanView.tsx` from its stub into a fully interactive plan-proposal screen.

The screen shows the agent-proposed sections (`{title, hypothesis}`) in an editable list.
Users can rename, drop, add, and reorder sections (all via `POST /plan/update` — backend-only,
no agent turn). The bottom bar sends free-form text to `POST /turn` (plan revision via the
agent). An "Accept plan & start building" button calls `POST /plan/accept` and advances.

## State management

**What stays local to PlanView:**
- The editable `sections` array is a _copy_ of what came in via props; it diverges from the
  server state as the user makes edits and reconciles on every `POST /plan/update` response.
- Edit-in-progress state (`editingId`, `editingTitle`, `editingHypothesis`) is purely local.
- Inflight flags and error string are local transient UI state.

**What comes from App.tsx via props:**
- `initialSections: Section[]` — seeded from `GET /state` on page load; PlanView initialises
  its local `sections` from this.

**What drives updates via SSE:**
- `plan.ready` — when the agent finishes a `/turn`-triggered plan revision, the new plan
  arrives via SSE. PlanView listens for `plan.ready` via a local `useSSE` call and resets
  its `sections` state from `event.sections`.

**App.tsx additions:**
- `plan: Section[]` added to `AppState` (initialised from `GET /state` response `plan` field)
- SSE handler for `plan.ready`: calls `api.getState()` (as a fallback) or reads directly from
  `event.sections`; updates `state.plan`
- `PlanView` receives `initialSections={state.plan}` as prop when stage is `"planning"`

**Why PlanView also subscribes to SSE directly (not just via props):**
- After a `/turn` call that revises the plan, the agent does a new turn and emits a new
  `plan.ready` event. The App's SSE handler will update `state.plan`, which would re-render
  PlanView with new props — but that would reset any in-progress local edits.
- Instead, PlanView subscribes directly to `plan.ready` to merge the server update at the
  right moment (e.g. only if the user is not mid-edit). This mirrors how `ProfileView`
  handles `profile.ready`.

## API call sequence

1. **Inline edit committed** → `POST /plan/update` with full sections array → on success:
   local `sections` confirmed; on error: `sections` reverts to last confirmed state.
2. **Bottom-bar submit** → `POST /turn` with text → 204 immediately; progress via SSE → on
   `plan.ready`, PlanView resets sections from event.
3. **Accept** → `POST /plan/accept` → 204 → App.tsx handles `stage.changed` → advances to
   BuildView (no PlanView action needed; stage routing handles this).

## Inline edit / reorder / drop / add mechanics

**Edit (title or hypothesis):**
- Each section row has an "edit" button (data-testid=`plan-edit-{id}`).
- On click: `editingId` set to section id; inputs pre-filled.
- On "save" click or Enter: compute new sections array with updated title/hypothesis; call
  `POST /plan/update`; clear `editingId`.
- On "cancel": clear `editingId` without saving.

**Drop:**
- Each row has a delete/drop button (data-testid=`plan-drop-{id}`).
- On click: filter section out of local array; call `POST /plan/update`.
- Minimum 1 section: disable drop if sections.length <= 1.

**Add:**
- "+ add section" button at the bottom (data-testid=`plan-add-section`).
- On click: append a new section with a generated id (`sec_new_<timestamp>`), empty title and
  hypothesis; set `editingId` to the new section; user fills in details; save commits it.
- The new section has `status: "queued"` as per the API contract requirement.

**Reorder:**
- Up/down arrow buttons on each row (data-testid=`plan-move-up-{id}`, `plan-move-down-{id}`).
- Swaps adjacent sections in local state; calls `POST /plan/update`.
- No drag-and-drop (out of scope for this story; too complex for an M-sized story).

**Calling `POST /plan/update`:** the full sections array is sent every time, as the API
contract specifies a full replacement. All sections must have `status: "queued"` per the
contract note ("Status must be `queued` for all items at plan stage").

## Bottom-bar behaviour

- Text input (data-testid=`plan-turn-input`) + submit button (data-testid=`plan-turn-submit`).
- Submit calls `POST /turn` with the text.
- While `isTurnInFlight`: input is disabled, submit disabled, spinner shown.
- On `plan.ready` SSE: sections reset, turn flight clears, input re-enabled.
- `POST /turn` failures: input preserved (user can retry), error shown in `plan-error`.
- `turn.error` SSE event: shown in the error banner for the stage (plan stage = plan revision
  failed).

## Vitest test plan (`PlanView.test.tsx`)

### Happy path

- Given `initialSections` with 3 sections, when rendered, then `plan-section-list` contains
  3 children with correct titles and hypotheses.
- Given edit is clicked on section 1, when title is changed and saved, then `api.postPlanUpdate`
  is called with the updated title.
- Given drop is clicked on section 2, when called, then `api.postPlanUpdate` is called without
  section 2 in the array.
- Given move-up is clicked on section 2, when called, then `api.postPlanUpdate` is called with
  sections 2 and 1 swapped.
- Given "+ add section" is clicked, when a title is entered and saved, then the new section
  appears in the list and `api.postPlanUpdate` is called.
- Given bottom-bar text is entered and submitted, when called, then `api.postTurn` is called
  with the text; input is disabled while in flight.
- Given `plan.ready` event arrives, when the turn is in flight, then sections reset and
  `isTurnInFlight` clears.
- Given accept is clicked, when `api.postPlanAccept` resolves, then call completes (App handles
  stage change).

### Error paths

- Given `api.postPlanUpdate` rejects, when an edit is saved, then `plan-error` shows an error
  message; local sections revert.
- Given `api.postTurn` rejects, when bottom-bar is submitted, then `plan-error` shows; input
  is re-enabled and preserved.
- Given `api.postPlanAccept` rejects, when accept is clicked, then `plan-error` shows and
  `plan-accept-btn` re-enables.
- Given a `turn.error` SSE event with `stage: "planning"`, when received, then `plan-error`
  shows the error.

### Edge cases

- Given `initialSections` is empty, when rendered, then only the "+ add section" row shows; no
  crash.
- Given a single section, when drop is clicked, then drop button is disabled (min 1 section).
- Given a section with very long title/hypothesis, when rendered, then the text is present in
  the DOM (no truncation that would break assertions).
- Given edit is in progress, when `plan.ready` arrives, then the new sections replace local
  state (edit is discarded to avoid stale data).
- Given bottom-bar text submitted with only whitespace, when Enter is pressed, then
  `api.postTurn` is NOT called (trim + guard).

### Null / missing inputs

- Given `initialSections` prop is `[]`, when rendered, then no section rows render; no crash.
- Given a section with `title: ""`, when rendered, then the row renders (empty string title is
  valid until user edits).
- Given `api.postPlanUpdate` resolves but returns `{ ok: false }` (unexpected), when handled,
  then the UI shows the error surface.

## Risks

1. **Reorder UX complexity.** The mockup shows "edit" and "×" buttons per row but no explicit
   up/down arrows. The story says "reorder" so up/down arrows are the simplest non-drag
   implementation. If TL prefers drag-and-drop, that would be an M→L size bump. Flagged here
   for plan review.

2. **plan.ready in both App.tsx and PlanView.** If both handle `plan.ready` via `useSSE`,
   there are two `EventSource` consumers. `useSSE` creates a new `EventSource` per call;
   each one subscribes independently. This means two SSE connections (or one per `useSSE`
   call site — need to verify). If `ProfileView` does the same (it does — confirmed in source),
   this is the existing pattern and is acceptable. Risk: multiple connections to `/api/events`.
   Mitigation: check that `useSSE` hooks are not creating duplicate EventSource instances.
   Actually, reviewing `useSSE.ts`: each call creates a new `EventSource`. So PlanView having
   its own `useSSE` means a second connection. This is the same pattern ProfileView uses; it
   was accepted in Night 1. However, TL should be aware this pattern compounds as more views
   do it. Documented here for review.

3. **Merge conflict with N2-S17.** Both stories add `plan: Section[]` to `AppState` and a
   `plan.ready` SSE handler in App.tsx. Resolution: whoever merges second removes the duplicate
   addition. Since N2-S17 is smaller (S vs M), it may merge first.

4. **Generated IDs for new sections.** The "add section" feature needs to generate a local id
   (e.g. `sec_new_<timestamp>`). The backend will re-assign IDs when it processes the
   `/plan/update`. The contract says sections sent to `/plan/update` should have an `id` field;
   if the backend rejects unknown IDs that don't match the existing plan, this will fail. Risk:
   **contract ambiguity** — the API contract says `/plan/update` accepts "full replacement plan
   array" but does not specify whether new IDs are accepted. **This may be a blocker if the
   backend rejects new IDs.** Noting here: if BE confirms that new IDs are accepted, proceed;
   if not, "add section" must be scoped down or the backend must be updated (out of FE lane).

5. **`POST /plan/update` must send `status: "queued"` for all sections.** The contract
   specifies this. PlanView must ensure the status field is always set to `"queued"` when
   building the update payload (the local copy starts with the status from `GET /state` which
   should already be `"queued"`, but we must not allow other status values to leak in).
