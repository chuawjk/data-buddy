## What this does

Implements `POST /plan/update` — accepts a full replacement plan array from the SPA,
validates it, persists to both `plan.json` and `state.json` atomically, and returns
`{"ok": true}`. No OpenCode call is made.

## Approach (ADR-014: full replacement semantics)

Per ADR-014: `POST /plan/update` accepts a full replacement plan array. New section
IDs are accepted; existing IDs preserve their status from the current plan.

Request body shape (from API_CONTRACT.html §1):
```json
{
  "sections": [
    { "id": "sec_01", "title": "...", "hypothesis": "..." }
  ]
}
```

Each section must have `id`, `title`, and `hypothesis`. `status` in the request is
optional and ignored for existing IDs (status is preserved). New section IDs get
`status="proposed"`.

Implementation steps:
1. Parse `body["sections"]` (must be a list, non-empty).
2. Validate each section has `id`, `title`, `hypothesis` fields.
3. Build a status-map from the current plan (`id → status`).
4. Merge: for each incoming section, preserve existing status if ID matches, else
   set `status="proposed"`.
5. Write `plan.json` to workspace (raw sections, no status) via `Orchestrator._write_plan_json()`.
6. `state_manager.update(plan=sections_with_status)`.
7. Return `{"ok": true}`.

## Error conditions

- `sections` key missing or not a list → 422 `invalid_request`
- `sections` is empty → 422 `invalid_request`
- A section missing required fields → 422 `invalid_section`

## Risk

- Need to access `workspace_root` in router: derive from `state_manager._path.parent`
  (same pattern used in `/export` and `/file`).
- `_write_plan_json` is a static method on Orchestrator — import it directly
  or replicate the logic. To avoid coupling router to orchestrator internals,
  replicate the atomic write inline (same pattern: tmp+rename).
