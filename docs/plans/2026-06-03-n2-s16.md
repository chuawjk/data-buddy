## What this story does

Builds out `BuildView.tsx` from its stub and introduces a new `SectionPane.tsx` component.
The screen shows the current section building live (via Activity Rail), then renders the
finished artefacts (code, chart, interpretation) once `section.proposed` arrives.
Accept/Drop buttons call backend-only endpoints; the bottom bar's redirect text calls `POST /turn`.

## State management

`BuildView` owns:
- `sections: Section[]` — hydrated from `GET /state` on mount + updated on `plan.ready` / `section.*` SSE events
- `activeSectionId: string | null` — the section currently building or proposed
- `redirectText: string` — bottom-bar input value
- `isSending: boolean` — POST /turn in flight

`SectionPane` receives `section: Section` as a prop and fetches its artefact files:
- `pyContent: string` — fetched via `GET /api/file?path=analyses/...` (text)
- `chartUrl: string` — rendered as `<img src="/api/file?path=charts/...">` directly (NOT fetched via api.getFile — that returns text; the img tag does its own HTTP GET with correct binary handling)
- `mdBody: string` — fetched via `GET /api/file?path=sections/...` (text); body extracted after frontmatter `---` delimiter

SSE events `BuildView` handles:
- `plan.ready` → refresh sections from `GET /state`
- `section.building` → mark active section as building, show spinner
- `section.proposed` → refresh section from `GET /state`, show artefacts
- `section.failed` → mark active section as failed (Night 3 error UI; for now just update status)
- `file.ready` → trigger `SectionPane` to re-fetch the relevant artefact

---

## Page-load hydration

On mount, `BuildView` calls `GET /state`. If sections are already `proposed` (browser refresh
mid-session), `SectionPane` renders directly from props — it does not require `section.proposed`
SSE to have fired.

## Approach

1. **BuildView** calls `GET /state` on mount to hydrate `sections`. Subscribes to SSE via `useSSE`
   hook already wired in `App.tsx`; receives events via prop callback or context.
2. **SectionPane** is a pure presenter receiving `section: Section`. Uses `useEffect` keyed on
   `section.py_path` and `section.md_path` to fetch artefact content only when paths change — avoids
   re-fetching on heartbeat-triggered re-renders.
3. **Chart rendering** uses `<img src="/api/file?path=...">` directly. The browser handles the
   binary GET; no JS fetch needed.
4. **Bottom-bar redirect** calls `api.postTurn({ text })`. On 400 `turn_busy`, shows `turn-busy-error`.
   On 204, sets `isSending=true` until next `section.proposed` or `section.failed`.
5. **Accept/Drop** call `api.postSectionAccept(id)` / `api.postSectionDrop(id)`. Both return 204.
   Update local section status optimistically.

## Risks

1. **Chart binary via img tag** — chart must use `<img src="...">` not `api.getFile()` since the
   latter returns text (would produce broken image). The `/api/file` endpoint must set correct
   `Content-Type: image/png` (covered by N2-S14).
2. **SectionPane artefact fetch deps** — `useEffect` must key on `py_path`/`md_path` not on the
   section object reference to avoid spurious re-fetches on every SSE heartbeat re-render.
3. **Hydration on refresh** — sections from `GET /state` may already be `proposed`; SectionPane
   must render from props immediately without waiting for SSE events.
4. **Stage routing** — `App.tsx` currently routes `building` to `BuildView` stub; needs `sections`
   prop wired through from App state.
5. **Section ordering** — sections should render in `plan.json` order, not arrival order.
