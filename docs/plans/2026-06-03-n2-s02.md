## What this does

Implement the planning turn: a new `backend/prompts/plan.py` module providing the plan prompt
template and JSON schema, plus the `session.idle` → `plan.ready` handling in the orchestrator.
On a successful plan turn, `plan.json` is written to the workspace and `plan.ready` is emitted.
On structured-output failure, `turn.error` is emitted.

## Architecture boundaries (confirmed)

- `backend/prompts/plan.py` is a pure data/string module — no imports from orchestrator, client, or httpx.
- Orchestrator calls `client.prompt(session_id, text, schema=PLAN_SCHEMA)` — narrow interface only.
- `plan.json` is written by the orchestrator via the filesystem (not by the client).
- `session.idle` is consumed internally; it is not forwarded to the SPA directly (per SSE_CONTRACT.md §2.5).

## Approach

### Part 1: `backend/prompts/plan.py`

#### `PLAN_SCHEMA`

Per WORKFLOW_ORCHESTRATION.md §Stage 3 and the backlog acceptance criteria, the schema is:

```python
PLAN_SCHEMA = {
    "type": "object",
    "required": ["sections"],
    "properties": {
        "sections": {
            "type": "array",
            "minItems": 3,
            "maxItems": 6,
            "items": {
                "type": "object",
                "required": ["id", "title", "hypothesis"],
                "properties": {
                    "id": {"type": "string"},
                    "title": {"type": "string"},
                    "hypothesis": {"type": "string"},
                },
            },
        }
    },
}
```

Note: `minItems`/`maxItems` are advisory constraints for the model. The acceptance criterion is
"3–6 entries"; validation at the handler level checks this range.

#### `build_plan_prompt(dataset, aim, profile, workspace_root)`

Mirrors `build_profile_prompt` in signature and structure. Returns a fully self-contained prompt:

```
Given the aim '<aim>' and the data profile below, draft a structured analysis plan.
Propose 3–6 sections. Each section has a unique id (e.g. 'sec_01'), a title, and a
hypothesis to test. Return the plan as structured JSON written to <plan_path>.

Profile:
<profile JSON>

Dataset: <csv_path>
```

Per ADR-011 (profile prompt must write to disk), the plan prompt must also explicitly instruct
OpenCode to write the JSON to `workspace/plan.json` — not just "return JSON". The structured
output format is still passed via the `format` parameter in `client.prompt()`, but the prompt
also directs the agent to write the file (belt-and-suspenders, per the ADR).

Per ADR-013, the `format` parameter is passed as `format.schema` (flat, no `json_schema` wrapper)
via `client.prompt(..., schema=PLAN_SCHEMA)` — the client already handles this translation.

#### Args

- `dataset: str` — filename of the CSV
- `aim: str` — user's stated analysis aim
- `profile: dict` — the parsed profile (from `state.json` or `profile.json`)
- `workspace_root: Path | str` — absolute workspace path

### Part 2: Orchestrator — `session.idle` → plan handling

Add `_handle_plan_idle` method, analogous to `_handle_profile_idle`:

1. Read current stage. If not `"planning"`, return early (ignore).
2. Check `workspace/plan.json` exists.
3. Read and parse JSON. If missing or parse error → emit `turn.error` (structured output failed).
4. Validate: `sections` is a list, each entry has `id`, `title`, `hypothesis`. If invalid →
   emit `turn.error`.
5. Check `3 <= len(sections) <= 6`. If out of range → emit `turn.error` with appropriate reason.
6. Write `plan.json` to `workspace_root / "plan.json"` (already there from step 2, but confirm).
7. Update `state_manager` with the plan: `self._state_manager.update(plan=sections)`.
8. Emit `plan.ready` with `{"sections": sections, "ts": <epoch_ms>}`.

**`turn.error` shape** (per SSE_CONTRACT.md §2.1):
```json
{"type": "turn.error", "stage": "planning", "reason": "structured_output_failed", "ts": <ms>}
```

Update `start_bus_listener`:

```python
elif event_type == "session.idle":
    stage = self._state_manager.get_state().get("stage", "")
    if stage == "profiling":
        await self._handle_profile_idle()
    elif stage == "planning":
        await self._handle_plan_idle()
```

This replaces the unconditional `_handle_profile_idle` call with a stage-aware dispatch. This is
the correct pattern: `session.idle` fires for every turn regardless of stage; the handler must
check which stage it's in.

**Note on N2-S01 interaction:** N2-S01 adds `profile.ready` → planning transition. N2-S02 adds
`session.idle` → plan output handling. These are independent changes to `orchestrator.py` on
separate branches. When both merge to `develop` (via squash), the combined `start_bus_listener`
will handle both events. The TL integration step reconciles any merge conflicts.

## Test plan (TDD — write tests first, then implementation)

### `backend/tests/unit/app/test_plan_prompt.py`

**Happy path:**
- `PLAN_SCHEMA` has `sections` array with `id`, `title`, `hypothesis` per item
- `build_plan_prompt(...)` contains the aim text
- `build_plan_prompt(...)` contains the profile JSON (serialised)
- `build_plan_prompt(...)` references the workspace plan.json path
- Schema `type` is `"object"` with `sections` as a required array

**Edge cases:**
- `build_plan_prompt(...)` with empty profile dict → does not crash; includes empty `{}`
- Schema has `minItems: 3`, `maxItems: 6` on the sections array

### `backend/tests/unit/app/test_orchestrator.py` — new tests

**Happy path:**
- Given `session.idle` in planning stage with valid `plan.json` (3–6 sections), when
  `_handle_plan_idle` runs, then `state_manager.update` called with `plan=sections`.
- Given valid output, when `_handle_plan_idle` completes, then `plan.ready` emitted with
  `sections` list and `ts`.

**Error paths:**
- Given `session.idle` in planning stage with missing `plan.json`, when `_handle_plan_idle` runs,
  then `turn.error` emitted with `stage="planning"`, `reason="structured_output_failed"`.
- Given `plan.json` with invalid JSON, when `_handle_plan_idle` runs, then `turn.error` emitted.
- Given `plan.json` with `sections` array of length 2 (< 3), when handler runs, then `turn.error`
  emitted.
- Given `plan.json` with `sections` array of length 7 (> 6), when handler runs, then `turn.error`
  emitted.
- Given `plan.json` missing required field `hypothesis` in one section, when handler runs, then
  `turn.error` emitted.

**Edge cases:**
- Given `session.idle` in `profiling` stage (not planning), when `_handle_plan_idle` is called,
  then it returns early without emitting.
- Given `plan.json` with exactly 3 sections → succeeds.
- Given `plan.json` with exactly 6 sections → succeeds.

**Null / missing inputs:**
- Given `session.idle` in planning stage with `plan.json` containing `null` sections field →
  `turn.error` emitted.

## Risks

1. **`session.idle` dispatch in `start_bus_listener`** — changing the unconditional
   `_handle_profile_idle()` call to a stage-aware dispatch is a modification to existing code.
   Existing N1 tests for `_handle_profile_idle` must continue to pass. The change is backwards
   compatible: profiling behaviour is unchanged (just guarded by stage check).

2. **Merge conflict with N2-S01** — both stories modify `backend/orchestrator.py`. N2-S01 adds
   `profile.ready` handling; N2-S02 updates `session.idle` handling. These are non-overlapping
   changes to `start_bus_listener`. TL integration resolves any conflict.

3. **`plan.json` already written by OpenCode** — the plan prompt instructs OpenCode to write the
   file directly (per ADR-011 pattern). The handler reads from disk. If OpenCode writes partial
   JSON (interrupted structured output), `json.loads` will raise and the handler emits `turn.error`.
   This is correct fail-safe behaviour.

4. **Profile data for plan prompt** — the plan prompt needs the profile. At the point
   `_run_plan_turn` fires (in N2-S01), the state already has `profile` populated from
   `_handle_profile_idle`. Read it from `self._state_manager.get_state().get("profile", {})`.
