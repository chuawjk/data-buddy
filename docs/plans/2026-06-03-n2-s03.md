# Plan: N2-S03 — Persist plan & section statuses

**Story:** N2-S03  
**Branch:** feat/n2-s03-persist-plan  
**Date:** 2026-06-03  
**Role:** BE  
**Size:** S

## Problem statement

When `_handle_plan_idle()` fires (triggered by `session.idle` in the planning
stage), it validates plan.json, injects a status on each section, and updates
`state.json`. Two things are missing:

1. **`plan.json` is not written back** by the backend after validation. The file
   that exists was written by OpenCode (structured output). The backend should
   write the canonical, normalised `plan.json` so downstream readers (GET /state,
   GET /export) always have a consistent file.
2. **Section status should be `"proposed"`** (the plan has been proposed to the
   user) rather than `"queued"` (sections awaiting build). The existing N2-S02
   code injected `"queued"`. N2-S03 corrects this: sections start as `"proposed"`
   at the plan level, meaning the plan is visible and awaiting user acceptance.
   After plan acceptance, sections transition to `"queued"` then `"building"`.

## Approach

### 1. Change status from `"queued"` to `"proposed"` in `_handle_plan_idle()`

In `backend/orchestrator.py`, the line:
```python
sections_with_status = [{**s, "status": "queued"} for s in sections]
```
becomes:
```python
sections_with_status = [{**s, "status": "proposed"} for s in sections]
```

### 2. Write `plan.json` with the canonical raw sections

After updating state, also write `workspace/plan.json` with the raw sections
(id, title, hypothesis — no status, per contract schema):
```python
plan_path.write_text(
    json.dumps({"sections": sections}, indent=2, ensure_ascii=False),
    encoding="utf-8"
)
```

The file is written atomically using the same tmp+rename pattern as
`state_manager.save()`.

### 3. Update affected test

The existing test `test_handle_plan_idle_injects_queued_status` expects `"queued"`.
This is updated to expect `"proposed"` and becomes
`test_handle_plan_idle_injects_proposed_status`.

## Tests (new file: `backend/tests/unit/app/test_persist_plan.py`)

Test file mirrors `backend/orchestrator.py` in the test tree at `backend/tests/unit/app/`.

Test cases:
1. **`test_plan_ready_writes_plan_json`** — happy path: valid plan.json on disk + planning
   stage → `_handle_plan_idle()` rewrites `plan.json` with canonical sections.
2. **`test_plan_ready_sections_have_proposed_status`** — each section in state.json
   has `status == "proposed"` after `_handle_plan_idle()`.
3. **`test_get_state_reflects_plan_with_statuses`** — after `_handle_plan_idle()`,
   `GET /state` (via `state_manager.get_state()`) returns the plan array with
   per-section statuses.
4. **`test_plan_json_written_atomically`** — write uses tmp+rename (tmp file cleaned up,
   final file present, content valid JSON).
5. **`test_plan_ready_missing_plan_json_no_write`** — missing plan.json → turn.error,
   no plan.json written.
6. **`test_plan_ready_invalid_json_no_write`** — malformed plan.json → turn.error,
   plan.json unchanged.
7. **`test_plan_ready_sections_exclude_status_in_json_file`** — plan.json on disk
   contains raw sections without a `"status"` field (status lives in state.json only).

## Scope

**In scope:**
- Changing status `"queued"` → `"proposed"` in `_handle_plan_idle()`.
- Writing canonical `plan.json` after validation in `_handle_plan_idle()`.
- Tests in `backend/tests/unit/app/test_persist_plan.py`.
- Updating the one existing test that asserts `"queued"`.

**Out of scope:**
- Plan edits / accept (`POST /plan/update`, `POST /plan/accept`) — own stories.
- Section build status transitions — N2-S07/S10/S11.
- `GET /state` HTTP layer — already implemented in router.py.
