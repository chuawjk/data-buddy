## Situation assessment

These three stories are small but tightly coupled: N3-S16 provides the test seam that drives
N3-S03's `turn.error` path deterministically, and N3-S02 provides the retry path that N3-S16's
recovery test exercises. Delivering them together on one branch avoids a sequencing problem where
N3-S16 depends on N3-S03 which depends on work whose test path is only exercisable via N3-S16.

### What already exists

- `Watchdog._handle_timeout()` already emits `turn.error` with
  `{"message": "turn timed out; session replaced"}`. The payload does NOT yet carry `stage` or
  `reason` fields required by the contract (`turn.error` payload: `{type, stage, reason, ts}`).
- `_run_plan_turn()` and `_run_profile_turn()` already catch exceptions and emit `turn.error` but
  again without `stage` or `reason`.
- `_run_section_turn()` emits `turn.error` without `stage` or `section_id`.
- The `POST /turn` router with an empty body re-fires the orchestrator call for the current stage,
  but today it requires `text` to be non-empty — an empty-body retry will return 422. The contract
  says "Retry button calls `POST /turn` with an empty body (re-fires the last prompt)".
- No `QA_FORCE_TURN_ERROR` env-var seam exists yet.

## Implementation approach

### N3-S03: Enrich `turn.error` payloads

The contract payload is `{type, stage, reason, ts}` where `reason` is one of
`"structured_output_failed" | "timeout" | "provider_error"`.

1. **`orchestrator._run_profile_turn()`** — catches `httpx.HTTPStatusError` (provider error) and
   any other exception (structured output exhausted). Emits:
   ```
   {"stage": "profiling", "reason": "provider_error" | "structured_output_failed", "ts": ...}
   ```

2. **`orchestrator._run_plan_turn()`** — same pattern for `stage="planning"`.

3. **`orchestrator._run_section_turn()`** — emits with `stage="building"` and additionally
   `section_id` (N3-S03 acceptance criterion: "stage/section reference").
   The contract payload does not list `section_id` as a top-level field in the `turn.error` card
   but the story says "with a stage/section reference". Adding it as an extra field is additive and
   safe; the FE story N3-S05 needs it to know which section's banner to show.

4. **`watchdog._handle_timeout()`** — reads current `stage` from `state_manager.get_state()` at
   timeout time. Emits with `reason="timeout"`. Since the watchdog does not know `section_id`,
   it omits it; the orchestrator can derive context from the current plan state when the FE retries.

### N3-S02: Retry a turn

**`orchestrator.retry_turn()`** — new public method called by the router on an empty-body
`POST /turn` or by the FE retry button. Logic:

1. Read current state: `stage`, `opencode_session_id`, plan, `retry_count` (new field in state).
2. Guard: if `retry_count >= MAX_RETRIES` (3), emit `turn.error` with
   `reason="max_retries_exceeded"` and return (bounded retry).
3. Increment `retry_count` in state.
4. Arm the watchdog.
5. Re-dispatch the exact prompt for the current stage/section:
   - `profiling` → `_run_profile_turn(session_id, prompt)`
   - `planning` → `_run_plan_turn(session_id, prompt)`
   - `building` → find the section with `status="building"` or `status="failed"` and
     call `start_build_section(...)`.

`retry_count` is reset to 0 whenever a stage transitions successfully (profile.ready, plan.ready,
section.proposed, stage.changed to building/done). This keeps retries bounded per-failure, not
per-session.

**`POST /turn` empty-body path** — the router currently requires `text`. Change: if `text` is
empty/missing AND the current stage has `last_error` set OR stage is appropriate, dispatch
`retry_turn()` instead. A simpler approach: just allow empty `text` and treat it as a retry
signal; the router checks whether `text` is empty and routes to `retry_turn()` vs the existing
stage-specific methods. This avoids storing `last_error` in state for the router.

**Targets the current/fresh session** — `retry_turn()` reads `opencode_session_id` from state
at call time, which will be the fresh session if the watchdog already swapped it. No special
handling needed.

### N3-S16: Forced turn-error hook (`QA_FORCE_TURN_ERROR`)

In `opencode_client.prompt()`: if `os.environ.get("QA_FORCE_TURN_ERROR") == "1"`, raise a
`RuntimeError("QA_FORCE_TURN_ERROR: simulated turn error")` immediately, before any HTTP call.
This drives the `except` clause in `_run_profile_turn` / `_run_plan_turn` / `_run_section_turn`,
which emit `turn.error`. After the forced error, the flag can be cleared and a subsequent retry
runs normally.

Acceptance criterion: "Given a retry after the forced error with the flag cleared, when it runs,
then the turn recovers." — the test sets `QA_FORCE_TURN_ERROR=1`, calls a turn method, asserts
`turn.error` is emitted, then unsets the flag, calls `retry_turn()`, and asserts the prompt is
sent normally (no error).

## Test coverage plan

### Happy paths
- `retry_turn()` in profiling stage re-sends the profile prompt.
- `retry_turn()` in planning stage re-sends the plan prompt.
- `retry_turn()` in building stage re-sends the section prompt for the failed/building section.
- `POST /turn` with empty body returns 204 and calls `retry_turn()`.
- `QA_FORCE_TURN_ERROR` cleared before retry → prompt sent normally.

### Error paths
- `retry_turn()` when `retry_count >= MAX_RETRIES` emits `turn.error` with `reason="max_retries_exceeded"`.
- `_run_profile_turn` on `httpx.HTTPStatusError` emits `turn.error` with `reason="provider_error"` and `stage="profiling"`.
- `_run_plan_turn` on generic exception emits `turn.error` with `reason="structured_output_failed"` and `stage="planning"`.
- `_run_section_turn` on exception emits `turn.error` with `stage="building"` and `section_id`.
- Watchdog timeout emits `turn.error` with `reason="timeout"` and current `stage`.

### Edge cases
- `retry_turn()` with no session ID raises/logs without crashing.
- `retry_turn()` in `setup` or `done` stage (no valid turn to retry) is a no-op or returns error.
- `QA_FORCE_TURN_ERROR=1` raises before making any HTTP call (no side effects on OpenCode).

### Null / missing inputs
- `retry_turn()` when plan is empty in building stage (no building/failed section).
- `retry_turn()` when state has no `profile` or `aim` (degenerate state).

## Risks and open questions

1. **`retry_count` in state vs. in-memory** — storing `retry_count` in `state.json` means it
   survives a backend restart, which is the right behaviour (a restarted backend should not
   lose the retry budget). Risk: the field is new in state and callers should tolerate `None`
   (treat as 0).

2. **`POST /turn` dual-mode (retry vs. new text)** — the contract says empty body triggers
   retry; non-empty body is a new instruction. This dual-mode is slightly surprising but matches
   the contract's stated routing-note. The router change is small: check if `text` is empty, then
   route to `retry_turn()`.

3. **`section_id` in `turn.error`** — not explicitly in the contract's turn.error JSON block but
   the story says "stage/section reference". Adding it as an optional extra field is additive.
   If TL disagrees, can be dropped — the FE can infer it from the plan's `building`/`failed`
   section.

4. **`QA_FORCE_TURN_ERROR` position** — placing the check in `opencode_client.prompt()` means
   it fires regardless of which stage calls it. This is the cleanest, least-invasive location.
   Alternative: check in each `_run_*_turn` method, but that duplicates the seam. The single-point
   approach follows the same pattern as `QA_FORCE_STALL` in the event subscription loop.

5. **MAX_RETRIES value** — the story says "bounded (no infinite auto-retry)" but doesn't specify
   the number. Using 3 (matching OpenCode's native `retryCount: 2` + 1 user-initiated retry)
   is reasonable. This is a soft constant, easy to change.
