## Overview

This story delivers the prompt builder for the section-build turn: `backend/prompts/section.py`.

The section build is the most complex agent turn in the system. Unlike profiling (Stage 2) and
planning (Stage 3), which use structured output (`format: json_schema`) to return machine-parseable
JSON, the section build uses **no structured output**. The section's structure is instead the
**file triplet** it produces on disk (ADR-005):

- `analyses/sec_NN_<slug>.py` â€” the Python analysis script
- `charts/sec_NN_<slug>.png` â€” the rendered chart
- `sections/sec_NN_<slug>.md` â€” interpretation text with YAML frontmatter

This story is scoped to the prompt builder and its tests only. The orchestrator wiring that
calls the prompt and the SSE event translation (section.building / section.proposed) belong to
N2-S05 and N2-S07 respectively.

## Design Decisions

### 1. No structured output (ADR-005, ADR-004)

Section builds must not include a `format` block in the `client.prompt()` call. The triplet
files are the structure. Forcing the whole section through a JSON schema would be awkward because:
- The chart artifact is a binary (`.png`)
- The analysis script may be hundreds of lines of code
- The frontmatter is deterministically parseable by the backend independently

The prompt builder therefore returns a text string to be called as:

```python
client.prompt(session_id, build_section_prompt(...))
# Note: schema=None â€” the default; no format block sent to OpenCode
```

This is the explicit departure from the profile and plan prompt builders.

### 2. File naming convention: `sec_NN_<slug>`

From the WORKFLOW_ORCHESTRATION.md Stage 4 template and the backlog acceptance criteria.

**NN â€” two-digit section index:**
- Zero-padded, 1-based: `sec_01`, `sec_02`, ... `sec_06`
- Derived from the section's position in `plan.json`'s `sections` array (1-indexed)
- The index is passed explicitly to the prompt builder as an integer; the builder formats it as
  `str(index).zfill(2)`

**slug â€” section title slug:**
- Derived from the section's `title` field in `plan.json`
- Transformation: lowercase, replace non-alphanumeric characters with underscores, collapse
  repeated underscores, strip leading/trailing underscores
- Maximum 40 characters after truncation to keep filesystem paths sane
- Example: `"Customer Churn by Tier"` â†’ `"customer_churn_by_tier"`

The `section_id` from `plan.json` (e.g. `"sec_01"`) provides the `NN` component. The slug is
derived by the prompt builder from the title at call time.

**Triplet paths produced:**
```
analyses/sec_01_customer_churn_by_tier.py
charts/sec_01_customer_churn_by_tier.png
sections/sec_01_customer_churn_by_tier.md
```

All paths are workspace-relative; the prompt supplies absolute paths using `workspace_root` so
OpenCode can find files even when `workspace/` is gitignored (same pattern as `profile.py` â€”
see ADR-011 lesson: always pass absolute paths).

### 3. Frontmatter schema for `sections/*.md`

The agent is instructed to write a YAML frontmatter block at the top of the `.md` file.
Required fields (minimum the orchestrator and N2-S09 parser depend on):

```yaml
---
section_id: sec_01
title: "Customer Churn by Tier"
hypothesis: "Higher-tier customers churn at lower rates."
chart: charts/sec_01_customer_churn_by_tier.png
---
```

Field specification:

| Field | Type | Required | Source |
|-------|------|----------|--------|
| `section_id` | string | Yes | from `plan.json` |
| `title` | string | Yes | from `plan.json` |
| `hypothesis` | string | Yes | from `plan.json` |
| `chart` | string | Yes | workspace-relative path to the `.png` |

These four fields are the minimum contract with N2-S09 (frontmatter parser), `GET /state` (for
section metadata), and the export builder. The agent may include additional fields; the parser
(N2-S09) must treat unknown fields as informational only.

The body of the `.md` (after the closing `---`) is free-form prose: the one-paragraph
interpretation of the chart and analysis findings.

### 4. Prompt template structure

Following the `build_profile_prompt()` pattern: a single function `build_section_prompt()` that
takes all required context as arguments and returns a fully self-contained prompt string. No
class, no state; the orchestrator passes everything in.

**Function signature:**

```python
def build_section_prompt(
    section_id: str,       # e.g. "sec_01"
    section_index: int,    # 1-based; used for zero-padded NN in filenames
    title: str,
    hypothesis: str,
    aim: str,
    dataset: str,          # just the filename, e.g. "customers_q3.csv"
    profile: dict,         # full profile.json contents
    plan: dict,            # full plan.json contents
    workspace_root: Path | str,
) -> str:
```

**Why pass profile and plan:** ADR-005 specifies that every prompt must be fully self-contained.
The agent has no dependency on its own conversational history. Profile (~2,000â€“4,000 tokens) and
plan (~200â€“400 tokens) are re-supplied on every section turn. This is the known, deliberate
redundancy described in ADR-005 Â§"Known inefficiency".

**Context for the agent:** The accepted sections before the current one are already on disk.
The plan JSON conveys the structure (section IDs, titles, hypotheses). The profile conveys
column types and flags needed to write sensible analysis code.

### 5. Prompt instructions: driving the file triplet

The prompt instructs the agent to perform these steps in order, making each step explicit:

1. Write a Python analysis script to `analyses/sec_NN_<slug>.py` using `apply_patch`.
   - Script must read the dataset from `data/<filename>.csv`
   - Script must produce a chart saved to `charts/sec_NN_<slug>.png` using matplotlib
   - Script must be self-contained (all imports included)
2. Execute the script via `bash`.
3. Write the section as a `.md` file to `sections/sec_NN_<slug>.md` using `apply_patch`.
   - `.md` must begin with a YAML frontmatter block containing the four required fields
   - Body (after frontmatter) is one paragraph interpreting the chart findings in the context
     of the hypothesis

The ordering is explicit because:
- The chart path must be known before writing the frontmatter (it is derived from the same slug)
- The interpretation should be written after the script runs so the agent can reference actual
  findings, not hypothetical ones

**Instruction emphasis on `apply_patch`:** Following ADR-011's lesson (profile turn), the
prompt explicitly instructs "use `apply_patch` for all file writes". This ensures the
`file.edited` events fire (the canonical detection signal per SSE_CONTRACT.md Â§D1) and avoids
the agent trying to shell-redirect output instead.

### 6. Context serialisation

Profile and plan are passed as Python dicts and serialised to JSON within the prompt builder
using `json.dumps(profile, indent=2)` and `json.dumps(plan, indent=2)`. This matches the
on-disk format and is the most unambiguous representation for the agent to read.

Profile is included in full (all column entries). Plan is included in full (all sections, not
just the current one), so the agent can see what has already been accepted and what is coming â€”
this avoids redundant analysis.

### 7. Redirect prompt (`build_section_redirect_prompt`)

WORKFLOW_ORCHESTRATION.md Â§Stage 4b specifies a distinct redirect prompt for mid-section
re-runs. The story scope specifies `backend/prompts/section.py` as the single touchpoint,
but the backlog says the redirect prompt lives in `backend/prompts/redirect.py` (N2-S12).

**Decision:** `section.py` exports only `build_section_prompt`. The redirect prompt belongs to
N2-S12 (`redirect.py`). This keeps scope tight: N2-S06 is the build turn only.

## No-structured-output design rationale

Restating for clarity because this is the key design choice of this story.

`client.prompt(session_id, text)` â€” no `schema` argument â€” means OpenCode receives no
`format` block in the `POST /session/:id/prompt_async` payload. The agent writes files freely
via `apply_patch`; `file.edited` events fire for each write; `session.idle` fires when the
turn is complete. The backend then checks for the presence of all three triplet files on disk
(N2-S07 / N2-S08) rather than parsing a structured response.

This avoids forcing a binary PNG through a JSON schema (impossible), keeping hundreds of lines
of Python code in a JSON string (fragile), and handling potential base64 encoding issues.
The trade-off is that the backend must inspect the filesystem at `session.idle` rather than
reading a single structured payload â€” but this is already the intended design (ADR-005).

## Risks and Mitigations

**R1 â€” Agent ignores file path instructions and uses relative paths**
The spike showed the agent uses `apply_patch` with absolute paths as provided.
Mitigation: supply all paths as absolute strings in the prompt (same pattern as profile.py /
ADR-011). If the agent uses a relative path, `file.edited` still fires with the absolute path
from OpenCode; detection in N2-S07/S08 reads from `workspace_root` directly.

**R2 â€” Agent writes PNG differently (e.g. shell redirect instead of apply_patch)**
The spike shows `file.edited` fires for any file write regardless of tool.
Mitigation: detection (N2-S07) uses `file.edited` as the canonical signal (SSE_CONTRACT.md
Â§D1), not a tool-name filter. The prompt does explicitly say `apply_patch` for all writes, but
`file.edited` is the backstop.

**R3 â€” Slug conflicts between sections**
If two sections have the same title (unusual but not impossible), they would produce the same
slug. Mitigation: the `section_id` (e.g. `sec_01`, `sec_02`) is part of the filename prefix, so
the full filename `sec_01_churn.py` and `sec_02_churn.py` remain distinct.

**R4 â€” Agent writes frontmatter incorrectly (wrong syntax or missing fields)**
Mitigation: the N2-S09 frontmatter parser is designed to fail safe â€” a section with malformed
frontmatter is flagged, not crashed. The prompt explicitly specifies the YAML format and the
four required fields with examples. This risk is accepted as residual (the agent's quality
controls for formatting are generally good).

**R5 â€” Profile/plan serialisation is too large for the prompt**
At the stated maximum of 6 sections and a wide dataset (e.g. 50 columns), the combined profile
+ plan JSON is estimated at ~6,000 tokens. Modern models handle this easily within a 128K
context window. ADR-005 Â§"Known inefficiency" explicitly accepts this cost.
Mitigation: none needed for the prototype. For production, a leaner context strategy would
be appropriate.

**R6 â€” N2-S07 depends on this story**
N2-S07 (section build events) depends on N2-S06. If N2-S06 is delayed, N2-S07 is blocked.
Mitigation: N2-S06 is scoped to just the prompt builder â€” it is not blocked by any other Night
2 story, only N1-S09 which is already merged. This story can start immediately.
