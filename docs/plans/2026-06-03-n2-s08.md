# Plan: N2-S08 — Detect failed section

**Date:** 2026-06-03

**Size:** S
**Depends on:** N2-S07

## Situation assessment

After reading `backend/orchestrator.py`, the `_handle_section_idle()` method already contains
the complete implementation of both the success and failure branches (merged as part of N2-S07's
PR). Specifically:

- If any of the three triplet files (`.py`, `.png`, `.md`) are missing after `session.idle`,
  `section.failed` is published with `section_id` and `reason: "missing_files"`.
- If all three are present, `section.proposed` is published.
- The method is only called from `start_bus_listener()` when `stage == "building"` and only on
  the `session.idle` event, which means mid-turn bash errors (which don't produce `session.idle`)
  never trigger it.

`test_section_events.py` already covers: triplet present → `section.proposed`, missing `.md` →
`section.failed`, missing `.png` → `section.failed`, and wrong-stage guards. These tests pass.

## What N2-S08 adds

Since the implementation is already present, this story's work is:

1. **Dedicated test file** `backend/tests/unit/app/test_section_failed.py` that explicitly
   documents and verifies the N2-S08 acceptance criteria:
   - AC1: `session.idle` without expected `.md` → `section.failed` with `section_id`
   - AC2: mid-turn bash errors (not idle) → `section.failed` is NOT emitted
   - Additional edge cases: all files missing, only `.py` missing, no building section in plan

2. **No implementation changes** are needed — `_handle_section_idle()` already satisfies the
   story's acceptance criteria as written.
