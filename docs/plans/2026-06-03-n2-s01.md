**Role:** BE

## What this does

Extend `backend/orchestrator.py` to add the planning stage to the state machine:

1. When `profile.ready` is accepted (i.e. the profiling turn completes and `_handle_profile_idle`
   emits `profile.ready`), the orchestrator transitions profiling → planning.
2. On entering planning, the plan prompt is auto-triggered via `client.prompt(...)` (narrow
   interface only) and `stage.changed` is emitted.

## Architecture boundaries (confirmed)

- Orchestrator never imports `httpx`. The only OpenCode call is `await self._client.prompt(session_id, text, schema=None)`.
- The client never imports the orchestrator.
- Planning prompt dispatch uses `asyncio.create_task()` (fire-and-forget), identical to profiling.
- `stage.changed` is emitted before the plan turn fires, so the SPA transitions to the planning
  view immediately.

---

## Approach

### Step 1: `profile.ready` → planning transition

The bus listener (`start_bus_listener`) already loops over all events. Currently it only handles
`session.idle`. Extend it to also handle `profile.ready`:

```python
if event_type == "profile.ready":
    await self._handle_planning_transition()
```

`_handle_planning_transition()` is a new method:

1. Read current stage from `state_manager.get_state()`. If stage is not `"profiling"`, return
   early (this event could theoretically fire at startup on a reload).
2. Call `self._state_manager.update(stage="planning")` — atomically persists the transition.
3. Emit `stage.changed` with `{"stage": "planning", "ts": <epoch_ms>}`.
4. If `self._client is not None` and session ID is present, schedule `_run_plan_turn` as a
   fire-and-forget `asyncio.create_task`.

### Step 2: `_run_plan_turn` helper

Same pattern as `_run_profile_turn`:

1. Call `self._build_plan_prompt(dataset, aim, profile)` to get the prompt text.
2. Call `schema = self._load_plan_schema()` — imports `PLAN_SCHEMA` from
   `backend.prompts.plan` with `ImportError` fallback (returns `None` if N2-S02 not yet merged,
   so N2-S01 remains independently deliverable).
3. Call `await self._client.prompt(session_id, prompt_text, schema=schema)`.
4. Catch any exception → publish `turn.error` on the bus.

### Step 3: `_build_plan_prompt` and `_load_plan_schema`

Follow the identical graceful-import pattern from `_build_profile_prompt`:

```python
def _build_plan_prompt(self, dataset: str, aim: str, profile: dict) -> str:
    try:
        from backend.prompts.plan import build_plan_prompt
        return build_plan_prompt(dataset, aim, profile, self._workspace_root.resolve())
    except ImportError:
        # N2-S02 not yet merged; minimal fallback
        return f"Draft an analysis plan for {dataset} with aim: {aim}. Return as JSON."

@staticmethod
def _load_plan_schema() -> Any:
    try:
        from backend.prompts.plan import PLAN_SCHEMA
        return PLAN_SCHEMA
    except ImportError:
        return None
```

### Step 4: `start_bus_listener` update

Add `profile.ready` handling alongside `session.idle`:

```python
async for envelope in subscription:
    event_type = envelope.get("type", "")
    if event_type == "session.idle":
        await self._handle_profile_idle()
    elif event_type == "profile.ready":
        await self._handle_planning_transition()
```

The `profile.ready` event is published by `_handle_profile_idle`, then consumed here by the
same listener. This is fine — the bus is fan-out (each subscriber gets its own queue), and the
orchestrator is its own subscriber. The ordering is: `session.idle` → `_handle_profile_idle` emits
`profile.ready` → the same listener loop's next iteration handles `profile.ready` → planning
transition fires. This is sequential within the single asyncio task, so no race.

## Test plan (TDD — write tests first, then implementation)

New tests appended to `backend/tests/unit/app/test_orchestrator.py`.

**Happy path:**
- Given `profile.ready` published on bus, when `start_bus_listener` processes it, then
  `state_manager.update` is called with `stage="planning"`.
- Given planning entered, when transition runs, then `stage.changed` with `stage="planning"` is
  published on the bus.
- Given planning entered with client + session, when transition runs, then `client.prompt` is
  called (fire-and-forget task scheduled).
- Given planning entered, when `stage.changed` fires, then it fires before the plan turn starts
  (ordering assertion: `stage.changed` event received before `client.prompt` mock is called).

**Error paths:**
- Given `_run_plan_turn` where `client.prompt` raises, when it runs, then `turn.error` is published
  on bus and the exception does not propagate out of the task.
- Given `profile.ready` arrives but current stage is not `profiling` (e.g. already `planning`),
  when `_handle_planning_transition` runs, then it returns early without emitting or calling
  `client.prompt`.

**Edge cases:**
- Given `profile.ready` with no session ID stored, when `_handle_planning_transition` runs, then
  `stage.changed` and `state update` still happen, but `client.prompt` is not called.
- Given `profile.ready` with `client=None`, when transition runs, then it completes without error.

**Null / missing inputs:**
- Given malformed `profile.ready` envelope (no `profile` key), when `_handle_planning_transition`
  runs, then it does not crash — profile is read from state (written by `_handle_profile_idle`),
  not from the event payload.

## Risks

1. **Bus listener ordering** — `profile.ready` is emitted by `_handle_profile_idle` inside the
   `session.idle` handler. The same listener task then picks it up on the next loop iteration.
   This is sequential and safe (single asyncio task, no concurrent writes). No lock needed.

2. **Stage guard on `_handle_planning_transition`** — important: `profile.ready` may be replayed
   from `GET /state` hydration in future Night 3 work. The stage guard (`if stage != "profiling":
   return`) prevents double-transition. Must be in the implementation.

3. **N2-S02 not yet merged** — the graceful `ImportError` fallback for `build_plan_prompt` and
   `PLAN_SCHEMA` means this story is independently deliverable. When N2-S02 merges, the real
   prompt and schema are picked up automatically.

4. **Watchdog** — the watchdog is armed in `re_profile()`. For the planning turn, the same
   pattern applies: `if self._watchdog is not None: self._watchdog.start_turn()` in
   `_run_plan_turn`. This should be included in the implementation to avoid a 60s hang if the
   plan turn stalls.
